       Resumo — Código Limpo, Capítulos 2, 4 e 5
       
       MARTIN, Robert C. Código Limpo: Habilidades Práticas do Agile Software. Alta Books, 2009.

### Capítulo 2 — Nomes Significativos
Nomes aparecem em todo lugar no software: variáveis, funções, classes, pacotes e arquivos. O autor defende que um bom nome precisa deixar claro por que aquele elemento existe, o que ele faz e como deve ser usado, sem depender de nenhum comentário para isso.
O capítulo traz orientações como usar nomes que revelem o propósito, evitar nomes que possam confundir, fazer distinções claras entre elementos parecidos e preferir nomes pronunciáveis e fáceis de localizar no código. Classes devem ter nomes com substantivos e métodos devem ter nomes com verbos.
A ideia central é que clareza vale mais do que brevidade. Um código que outra pessoa consegue ler sem dificuldade é mais útil do que um cheio de abreviações que só fazem sentido para quem o escreveu.

### Capítulo 4 — Comentários
O autor trata comentários como algo a ser usado com bastante cuidado. Para ele, um comentário muitas vezes indica que o código não ficou claro o suficiente por si só. E quanto mais tempo passa, maior a chance de um comentário estar desatualizado e passando informações erradas.
Existem situações em que comentários são justificados: obrigações legais, explicação do motivo de uma decisão específica, alertas sobre consequências e anotações de tarefas pendentes. Já comentários que apenas repetem o que o código já diz, trechos de código desativados deixados no arquivo e históricos de alterações são apontados como prejudiciais.
A orientação principal é escrever um código claro o bastante para não precisar de explicação, usando comentários só quando não há outra forma de passar a informação.

### Capítulo 5 — Formatação
A formatação é tratada no livro como parte da comunicação, não só como uma questão visual. O autor compara o código a um artigo de jornal: os conceitos mais gerais aparecem primeiro e os detalhes vão surgindo conforme a leitura avança.
Na parte vertical, a orientação é separar blocos de código com linhas em branco quando representam ideias diferentes, e manter funções relacionadas próximas. Arquivos menores são mais fáceis de entender. Na parte horizontal, linhas curtas e uso adequado de espaços ajudam na leitura.
O capítulo também fala sobre trabalho em equipe: quando se programa junto com outras pessoas, o estilo de formatação precisa ser definido em grupo e seguido por todos. Consistência entre os arquivos do projeto é mais importante do que preferências individuais.

### Conclusão
Os três capítulos tratam de aspectos diferentes, mas chegam ao mesmo ponto: o código é escrito uma vez e lido muitas vezes. Nomear bem, comentar com cuidado e manter uma formatação consistente são práticas que facilitam o trabalho de quem vai dar manutenção no código depois.
Escolher bons nomes, evitar comentários desnecessários e manter a formatação consistente são atos de respeito com quem vai ler o código depois (o que vale para o próprio criador do código).
